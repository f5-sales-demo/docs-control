This page covers the procedure to enroll a new repository in the docs-control governance system.

## Prerequisites

- Org membership in `f5xc-salesdemos`
- `REPO_SETTINGS_TOKEN` and `REPO_SYNC_TOKEN` secrets configured as organization-level secrets (or added to the new repo individually)
- Access to the `ghcr.io/f5xc-salesdemos/docs-builder` container image

## Steps

### 1. Add to downstream-repos.json

Add the repo's `owner/repo` string to `.github/config/downstream-repos.json` in docs-control. This registers it for dispatch and enforcement.

### 2. Optionally add to docs-sites.json

If the repo will publish documentation, add an entry to `.github/config/docs-sites.json` with:
- `label` -- human-readable site name
- `url` -- URL to the site's `llms-full.txt` endpoint
- `description` -- short description used in the generated README

If no entry is added, the README generator falls back to a capitalized repo name and the GitHub API description.

### 3. Install caller workflows

Copy the three caller workflow templates from `workflows/` in docs-control into the new repo's `.github/workflows/` directory:

- `enforce-repo-settings.yml` -- triggers enforcement and file sync
- `github-pages-deploy.yml` -- triggers docs build and deploy
- `require-linked-issue.yml` -- enforces PR-to-issue linking

These files are also synced automatically by the [file sync workflow](./file-sync/), but installing them manually bootstraps the process.

### 4. Trigger enforcement

Run the enforcement workflow manually in the new repo:

```bash
gh workflow run enforce-repo-settings.yml --repo f5xc-salesdemos/<repo-name>
```

This applies all [repository settings](./settings-enforcement/), creates any missing governance files, and opens a sync PR if needed.

### 5. Verify settings

Confirm that enforcement succeeded:

```bash
gh run list --repo f5xc-salesdemos/<repo-name> --workflow enforce-repo-settings.yml --limit 1
```

Check that branch protection, Actions permissions, and Pages are configured correctly:

```bash
gh api repos/f5xc-salesdemos/<repo-name>/branches/main/protection \
  --jq '.required_status_checks.contexts'
```

### 6. Verify docs site

If the repo has a `docs/` directory, confirm the docs site is accessible after the first successful deploy:

```bash
curl -sf "https://f5xc-salesdemos.github.io/<repo-name>/" \
  && echo "OK" || echo "FAIL"
```

## Troubleshooting

### Chicken-and-egg problem

The enforcement workflow requires the caller workflows to be installed, but the file sync that installs them needs the enforcement workflow to run. Break the cycle by manually installing the three caller workflows first (Step 3), then triggering enforcement (Step 4).

### Pages not building

Ensure:
- The repo has a `docs/` directory with at least an `index.mdx` file
- GitHub Pages is enabled with `build_type: workflow` (enforcement handles this)
- The `GITHUB_TOKEN` has `pages: write` and `id-token: write` permissions

### Stale sync branches

If a `governance/sync-managed-files` branch exists from a previous run, the file sync workflow will attempt to update it. If it's stale, delete it manually:

```bash
gh api repos/f5xc-salesdemos/<repo-name>/git/refs/heads/governance/sync-managed-files \
  --method DELETE
```
